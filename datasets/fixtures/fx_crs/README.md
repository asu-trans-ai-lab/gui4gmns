# fx_crs — CRS reprojection fixture (D9 geographic correctness)

A 3-node / 2-link network whose `node.csv` coordinates are real **UTM Zone 12N (EPSG:26912)** meters
(not lon/lat), paired with `crs.txt = EPSG:26912`, so the generator's `--basemap`-driven reprojection
path is exercised. The three UTM points were computed with `pyproj` from three known WGS84 points near
Tempe, AZ, so the *correct* reprojected answer is known exactly.

| node | UTM12N x, y (raw input) | expected WGS84 lon, lat |
|---|---|---|
| 1 | 412610.172, 3698710.293 | -111.9400, 33.4242 |
| 2 | 413075.028, 3698706.103 | -111.9350, 33.4242 |
| 3 | 412613.980, 3699131.613 | -111.9400, 33.4280 |

## What this fixture found

Generating it exposed a **real D9 (geographic correctness) bug in `ai-gen/gui4gmns.py`**: the CRS
transformer was only ever applied to **link** geometry, never to **node** coordinates. `nid` (and the
embedded `D["nodes"]` layer used for node-dot rendering) held the raw, unprojected UTM values while
links were correctly reprojected to lon/lat — so with any `crs.txt`, the node markers and the road
lines would render in two completely different coordinate spaces (node dots off by ~400,000 in x /
~3,700,000 in y from where the roads actually are).

**Fixed**: the CRS transformer is now built *before* the node loop, and node coordinates are projected
when `nid`/`D["nodes"]` are constructed. Because `nid` is reused for the link-geometry fallback and
`zone_pts` feeds the OD-desire-line centroid calc, both of those call sites were updated to stop
re-applying `proj()` to already-projected coordinates (which would have silently double-transformed
them into nonsense). Also fixed a cosmetic log bug: `crs.txt` containing `EPSG:26912` printed
`reprojected EPSG:EPSG:26912` (doubled prefix).

**Not fixed here (separate, larger scope — flagged, not silently built):** the auto-exported portals
(kepler/deckgl/qgis/kml) are generated by `_export_portals()`, which reads the folder independently via
`exporters/gmns_to_viz.py`'s own `read_gmns()` — a completely separate code path from `load()` that has
no `crs.txt` support at all. So a projected-CRS dataset gets a **correctly reprojected interactive
dashboard** but **raw-unit (broken) portal exports** — kepler/deckgl/kml specifically require lon/lat to
work. Needs its own fix in `exporters/gmns_to_viz.py`, out of scope for this fixture.
